● Good question. The simple Registry I described — a URL→desc map — works for today but has a glass ceiling. Let me think about what "upgrade"
  means here.                                                                                                            

  The pipeline will eventually want:

  - Same image re-uploaded → instant cache hit (content-based, not URL-based)
  - Richer metadata beyond descriptions: color palette, tags, embedding vectors for similarity search
  - Batch operations with progress: scan 12 images, show "5 of 12 done"
  - Multiple scan strategies: today it's Gemini 2.5 Flash, tomorrow maybe a different model per image type
  - Session vs persistent: descriptions that survive page refresh (IndexedDB) vs temporary ones

  So the storage shape should handle all of that now, even if we only populate desc today.

  ---
  Proposed shape:

  Registry entry:
  {
    hash: "a1b2c3...",       // SHA-256 of base64 data (content-addressable)
    desc: "tall woman...",   // text description (what we use today)
    scannedAt: 1715432000,   // timestamp
    model: "gemini-2.5-flash",
    // future:
    palette: null,           // ["#ea5823", "#c7c7c7", ...]
    tags: null,              // ["portrait", "outdoor", "golden-hour"]
    embedding: null          // float[] for similarity search
  }

  API surface:

  Registry.ensure(base64, context)        → Promise<entry>   // scan if needed
  Registry.ensureAll([{url, context}])    → Promise<entry[]>  // batch with progress
  Registry.get(url)                       → entry|null
  Registry.clear({since})                 // selective clear by timestamp
  Registry.clearAll()                     // full wipe
  Registry.onProgress(fn)                 // "3 of 12 scanned"

  Key insight — hash as primary key, URL as lookup:

  The hash identifies the image content. Two different data URLs with the same visual content produce the same hash → one scan serves both. The
  URL is just an index for fast lookup. A url→hash map sits alongside the hash→entry map.

  This matters when a user uploads the same reference image in two different projects, or when the same module image gets re-uploaded after
  removal. Currently those would be separate VisionScan calls. With hashing, the second upload hits cache instantly.

  For Keep OFF: Registry.clear({since: thisGenerateStart}) removes only entries from this run. Previous scans survive. Faster and more precise
  than wiping everything.

  For scan timing: Registry.ensureAll() at upload time (On Load) or at generate time (On Generate). Same function, different call site.

  ---
  This is future-proof without being overbuilt. The entry shape has 5 fields today (we populate 2), the API has 5 methods (we use 3). Adding
  palette extraction later means changing the VisionScan prompt and populating one more field — no schema migration, no refactor.

  Worth building this instead of the simpler map? Or keep the simple version with the entry shape ready for extension?
